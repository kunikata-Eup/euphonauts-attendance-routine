# freee勤怠コンプライアンスチェック Routine 仕様書

このファイルは、Claude Code RoutineのPromptから参照される「実行仕様の本体」です。
Routine側のPrompt欄は短いブートストラップ文のみとし、実際の判定基準・フォーマット等は
すべてこのファイルに保存しています（Routineのプロンプト欄が何らかの理由で途中で
途切れて伝わる事故が過去に発生したため、単一障害点をなくす目的でこの構成にしました）。

このファイルを更新した場合は、必ずコミット・pushしてください。次回のRoutine実行時に
最新の内容が反映されます。

---

あなたは株式会社Euphonautsの労務コンプライアンスチェック担当です。
リポジトリ内にコミットされている、手動エクスポート済みのfreee「日次勤怠一覧」CSVを
読み込み、下記9項目をチェックして、結果をSlackの指定チャンネル
（#hr_勤怠アラート_自動通知）に日本語で投稿してください。

【最重要・厳守事項】
0. **Slack投稿だけでなく、実行中の思考過程・進捗報告・最終報告（routine_summary）も
   すべて日本語で出力すること。** 英語での出力は禁止する（コード自体や、CSVの列名・
   ファイルパスなど固有名詞はそのままでよいが、説明文・報告文は日本語で書くこと）。
1. **freee MCP（API）は一切呼び出さないこと。** 必要なデータはすべてリポジトリ内の
   CSVファイルから読み込む。これがこの方式の目的（クレジット最小化）である。
2. 必要な月のCSVがリポジトリに存在しない場合は、無理に代替手段（freee API等）を
   使わず、後述の「CSVが無い場合の処理」に従ってSlackにリマインドを送って終了する。
3. 分析はPython（pandas等）のスクリプトで一括処理し、行ごとにLLMが目視確認・
   コメントすることはしない（トークン消費を抑えるため）。
4. **クレジットを最小限に抑えることを常に最優先で意識する。** 迷ったら呼び出し回数・
   トークン消費が少ない方法を選ぶこと。
5. **「CSVには入力ミスや欠損があるかもしれない」という前提でチェックする。**
   下記「0. データ品質の自動チェック」を省略しないこと。
6. 実行中に想定外の事象（パース失敗、フォーマットの想定違い、列名の変化など）が
   発生した場合は、`docs/freee_csv_notes.md`に「発生した問題」「原因」「今後の対策」を
   追記してコミットする。次回実行時は必ずこのファイルを読んでから作業を始め、
   同じ失敗を繰り返さないようにする（このファイルを学習ログとして育てていくこと）。

## CSVフォーマットの既知仕様（判明済み・再調査不要）

- ファイル名パターン：`日次勤怠一覧_月末締め_翌月末日払い__{year}_{payment_month}.csv`
  （プレフィックス部分は締め日設定により多少変わる可能性があるため、末尾の
  `_{year}_{payment_month}.csv` 部分で一致するファイルを探すこと）
- `{payment_month}` は実労働月+1（支給月）。例：2026年6月の実働データは
  ファイル名の月部分が「07」になる（`..._2026_07.csv`）。年またぎ（12月→1月）の
  繰り上がりに注意。
- 主要列（列名は正確にこの通り）：
  `氏名`, `従業員番号`, `部門`, `日付`, `曜日`, `勤務日種別`, `勤怠種別`,
  `出勤時刻`, `退勤時刻`, `所定内労働時間`, `法定内残業時間`, `時間外労働時間`,
  `法定休日労働時間`, `深夜労働時間`, `遅刻時間`, `早退時間`, `休憩時間`,
  `総勤務時間`, `有休時間 - 合計`, `勤怠メモ`
- `勤務日種別`列の値は `所定労働日` / `所定休日` / `法定休日` などで休日判定に使う。
- `深夜労働時間`列が `00:00` 以外なら、その日は深夜勤務に該当する（退勤時刻から
  自前で計算し直す必要はない。ただし、22:00ちょうど退勤は深夜労働として計上されない
  仕様である可能性があるため、`深夜労働時間`が0でも退勤時刻が22:01以降なら
  念のため深夜勤務として拾うこと）。
- `総勤務時間`列がその日の実働時間（休憩控除後）。長時間労働（10時間以上）の判定に使う。
- `時間外労働時間`列がその日の残業時間。週次・月次の残業集計に使う。
- 時間は `HH:MM` 形式の文字列。`00:00`は「時間なし」を意味し、欠勤や未入力とは異なる
  （出勤時刻・退勤時刻が空欄かどうかで打刻漏れを判定する）。
- ファイルの文字コードはUTF-8 with BOM（先頭に`﻿`が付く場合がある）。pandas等で
  読み込む際は`encoding='utf-8-sig'`を指定すること。

## 手順

### ステップ0：パイプライン起因の異常を検知する軽量な事前検証（最優先）
このRoutineは「Cowork（毎朝7:00・freee側でエクスポートをトリガーするのみ）→
Windowsタスクスケジューラ（毎朝7:30・ダウンロードフォルダから`data/manual_csv/`へ
移動）→ このRoutine（毎朝8:00・分析）」という3段構成の最後の工程にあたる。
1と2は独立して動いているため、8:00時点で`data/manual_csv/`のファイルが
「本当に今日の最新データか」は保証されていない。

**この事前検証は、9項目の詳細分析とは別に、必ず先に行う軽量なチェックに限定する
こと（クレジット節約のため、ここでは深い分析をしない）。** a〜eのいずれかで異常が
見つかった場合は、**9項目チェックには一切進まず**、該当する内容だけをSlackに
1回投稿して終了する。すべて正常な場合のみステップ1（対象期間の確定）に進む。

異常を報告する際は、必ず**「⚠️ データ取得異常」**という接頭辞を使うこと。これは
後述の9項目チェックで検出する**「🚨 勤怠コンプライアンス違反」**（労務上の実際の
問題）とは性質が異なる（＝取得パイプラインの不具合であり、労務問題ではない）ため、
Slack上で混同しないよう明確に区別すること。

#### 0-x) 作業ブランチの確認（最優先・他のどのステップより先に実行）
このRoutineが参照するリポジトリのクローンが、正しいブランチ（GitHub上の
デフォルトブランチ、＝CSVがpushされる先）を見ているとは限らない。過去に、
ローカルの`move_csv.bat`側のpush先ブランチと、Routine側のクローンが参照する
ブランチが食い違いかけた事例があったため、必ず以下を実行してから他の処理に進む。
（gitコマンドのみで完結する軽量な確認のため、クレジット消費はごくわずか。）

1. `git remote show origin` を実行し、リモートのデフォルトブランチ名（HEAD branch）
   を取得する。
2. `git branch --show-current` で、現在の作業ブランチ名を取得する。
3. 1と2が一致しない場合、以下を実行してデフォルトブランチに切り替える。
   ```
   git fetch origin
   git checkout [デフォルトブランチ名]
   git pull origin [デフォルトブランチ名]
   ```
   **【フォールバック】** 実行環境の制約でブランチの切り替え自体ができない場合
   （例：Claude Code on the webなど、セッションに割り当てられた作業ブランチが
   固定されている環境）は、無理に切り替えようとせず、代わりに
   `git rev-parse HEAD`（現在のブランチ）と`git rev-parse origin/[デフォルトブランチ名]`
   （デフォルトブランチ）の**コミットハッシュを比較**する。両者が一致していれば
   「ブランチ名は違うが、参照している内容は同じなので鮮度・整合性に問題なし」と
   判断し、通常通り処理を続行してよい。一致していない場合のみ、後続の分析を
   保留し、ハッシュが不一致である旨を最終報告に明記する。
4. 切り替え、またはハッシュ比較による代替検証を行った場合は、その旨を最終報告
   （routine_summary）に一言だけ記録する。
   例：「作業ブランチを[旧ブランチ名]から[デフォルトブランチ名]に切り替えました」
   例：「ブランチ名は不一致だがHEADコミットが一致していたため、コミット比較で
   代替検証した」
5. この確認・切り替え処理は**毎回必ず実行し、省略しないこと**（move_csv.bat側で
   push先ブランチが固定されていても、Routine側の環境が古いブランチのまま
   セットアップされている可能性があるため、双方で独立してチェックする）。

#### a) 更新日（鮮度）チェック
**この鮮度チェックは「まだ確定していない可能性がある月」のCSVにのみ適用する。**
「確定済みの月」の定義は以下の通り（ステップ1・ステップ2の「月初5営業日以内は
前月分も対象」というルールと整合させること）。

- **今月（実労働月として進行中の月）**：常に「未確定」＝鮮度チェック対象
- **先月（直前の暦月）**：
  - 今月の**最初の5営業日以内**（土日を除く実行日ベースでカウント）である場合、
    先月分もまだ確定していない可能性がある（freee側で遅れた修正が入ることが
    あるため）ので、**鮮度チェック対象とする**
  - 今月の**6営業日目以降**であれば、先月分は確定済みとみなし、鮮度チェックは
    行わない（単に存在していればよい）
- **先々月以前**：常に確定済みとみなし、鮮度チェックは行わない

**【判定方法・確定版】判定根拠はCSVファイル自体のgit差分ではなく、
`data/manual_csv/move_log.txt`の本日分のパイプライン実行記録を使う。**
（過去にgit差分ベースで判定していたところ、「確定済み月はfreee側の内容が
変わらずgit差分ゼロになる」という正常なケースを異常と誤検知したため変更した。）

1. `data/manual_csv/move_log.txt`を**`cp932`（Shift_JIS）でデコードして**読み込む。
   `utf-8`のままデコードすると日本語部分が文字化けし、誤診断の原因になる
   （過去に実際発生した）。
2. ログの最新エントリ（`===== 日時 =====`区切りの最後のブロック）のタイムスタンプが
   **本日の日付**であるか確認する。本日でなければ「未実行」と判定し、下記の中断処理
   に進む。
3. そのエントリ内の**`move`コマンドの生出力部分**（後段の「残存ファイルなし」判定
   ではなく、実際に移動処理を行った際の出力）に、**今回対象となる各支給月ファイル名
   （年_月の部分）が実際に含まれているか**を、未確定と判定された月の数だけ確認する。
   - 【注意】後段の「移動処理: Downloadsフォルダに残存ファイルなし（正常）」という
     行だけでは判定しないこと。この行は「Coworkが今日エクスポートに失敗し、
     Downloadsに何も生成されなかった場合」でも同様に「正常」と表示されてしまう
     抜け穴があるため、必ず`move`コマンド自体の生出力（成功時は移動したファイル名が
     個別に出力される）まで見て、対象ファイル名が実在するかを確認する。
4. 未確定と判定された月のうち、1つでも「本日のログに、その支給月ファイルの移動
   成功記録が見当たらない」場合は、以下を投稿して中断する（9項目チェックには進まない）。
```
⚠️ データ取得異常：move_log.txtの本日分の移動記録が確認できません
[支給月ファイル名] の本日分の移動成功記録が見当たりません。
パイプライン（Cowork／タスクスケジューラ）の確認が必要です。
```
5. すべての未確定月について本日分の移動成功記録が確認できれば「パイプライン正常」
   とみなし、CSV自体の内容差分の有無（gitコミットがあったか）は問わずに次の
   ステップへ進む（確定済み内容で差分ゼロなのは正常なケースであり、異常としない）。

#### b) 行数（想定従業員数）チェック
対象CSVの行数を`data/csv_pipeline_state.json`に記録された前回実行時の行数と比較する
（このファイルが無ければ今回の行数を記録して正常とみなし、次回以降の比較に使う）。
前回比で**±20%以上**乖離していれば異常とみなし、以下を投稿して中断する。
```
⚠️ データ取得異常：CSVの行数が想定と乖離しています
今回: [件数]件 / 前回: [件数]件
CSV取得パイプラインで一部データが欠落・重複している可能性があります。
```

#### c) 日付範囲チェック
CSVの`日付`列が、ファイル名から想定される対象カレンダー月（支給月-1）の日付のみで
構成されているか確認する。対象月以外の日付（前月・翌月）が混入していれば異常とみなし、
以下を投稿して中断する。
```
⚠️ データ取得異常：CSVの日付範囲が想定と異なります
想定対象月: [年月] / 実際に含まれる日付範囲: [最小日付]〜[最大日付]
freee側の月またぎ処理や、誤ったファイルが配置された可能性があります。
```

#### d) 文字コード・文字化けチェック
`encoding='utf-8-sig'`で読み込めるか、また代表的な列（氏名等）に文字化けを示す
記号（`�`等）や異常なバイト列が含まれていないかを簡易的に確認する。異常があれば
以下を投稿して中断する。
```
⚠️ データ取得異常：CSVの文字コードに問題がある可能性があります
[検出した内容の概要]
```

#### e) ファイル欠落チェック（中断はしない・注記のみ）
ステップ1で必要と判定される2ファイル（当月分・前月分、月初5営業日以内の場合のみ
前月分も必要）のうち、**片方のみ**が見つからない場合は、a〜dのような中断はせず、
**見つかった方のみで処理を続行**する。ただし、最終的なSlack投稿の冒頭に、
以下のように欠落を明記すること。
```
⚠️ データ取得異常：[支給月]分のファイルが見つかりませんでした（片方のみで分析）
```
両方とも見つからない場合は中断し、以下を投稿する。
```
⚠️ データ取得異常：分析に必要なCSVが両方とも見つかりません
```

a〜dのいずれかで異常となった場合、`data/csv_pipeline_state.json`の行数記録は
更新しない（異常なデータを「正常な前回値」として記録してしまわないため）。
すべて正常だった場合のみ、今回の行数等を`data/csv_pipeline_state.json`に記録し、
次のステップに進む。

### ステップ1：対象期間を確定する
**「実行日の前日」を終点とする過去7日間**（実行日-7日 〜 実行日-1日）を対象期間とする。
（例：7/6に実行する場合、対象期間は6/29〜7/5）
※以前は「直近の完了週（月〜日）」という曜日ベースの計算をしていたが、算出ミスが
発生したため、日次実行にも自然に対応できるこのローリング方式に変更した。

この対象期間の`start_date`・`end_date`は**この時点で変数として1回だけ確定し、
以降のステップやSlack投稿では、この変数をそのまま使い回すこと。別の場所で
再計算したり、ハードコードし直したりしないこと**（表示ラベルと実際の集計対象が
ズレるミスを防ぐため）。

対象期間の日付がまたがる暦月をすべて洗い出し、それぞれの実労働月に対応する
支給月（+1）を計算して、今回読み込むべきCSVファイル名の候補を確定する
（通常は1つ、月をまたぐ場合のみ2つ）。

### ステップ2：必要なCSVがリポジトリに存在するか確認する
`data/manual_csv/` 配下を確認し、ステップ1で確定したファイル名（末尾一致）が
存在するか確認する。ファイルの過不足の扱いは、上記「ステップ0のe）ファイル欠落
チェック」の方針に従う（**両方とも見つからない場合のみ中断し、片方だけ見つかった
場合は見つかった方のみで処理を続行し、欠落をSlack投稿冒頭に明記する**）。

### ステップ3：CSVを読み込む
必要なCSVファイルをpandasで読み込む（`encoding='utf-8-sig'`）。複数月にまたがる場合は
結合する。以下の4名は対象外として除外する（氏名の姓名間スペースの有無や全角/半角の
揺れがあっても正しく除外できるよう、比較前に空白を除去してから照合すること）。
- 國方翔冴
- 飯村拓也
- 関口雄大
- 川上貴大

さらに、以下は**育休・休職等により一時的に対象外**とする（復職したら対象に戻す必要が
あるため、このリストは定期的に見直すこと。復職時期が分かっている場合は
`docs/freee_csv_notes.md`にメモしておき、その時期以降は自動的に対象から外れるように
判断すること）。
- 諸田秀祐（育休中）
- 朝長京香（育休中）
これらの従業員は、打刻漏れ等がCSV上に大量に出ていても「育休中のため正常」であり、
一切のチェック対象・Slack報告対象としない。

### 0. データ品質の自動チェック（ミスがある前提の防御的チェック。該当時のみ報告）
以下をスクリプト内で自動検証し、異常が見つかった場合のみ`result_summary.json`に
「data_quality_issues」として記録する（異常がなければ何も出力しない＝Slackにも表示しない）。
- CSV内のユニーク従業員数（対象外4名除外後）を、直近の別月CSVと比較する。
  **人数が増えるのは正常な新規採用なので報告不要。** 前月に存在した従業員（氏名＋
  従業員番号のペア）が今月のCSVに見当たらない場合は、**「異常」や「データ品質の問題」
  としてではなく**、退職・異動等による自然な変化である可能性が高いため、
  `data_quality_issues`には含めず、別枠の「退職者確認メモ」として氏名を淡々と
  列挙するだけにとどめる（原因を断定したり、問題扱いしたりしない）。
- 日付列が対象月の全日をカバーしているか（歯抜けがないか）
- 出勤時刻が退勤時刻より後になっている、時間形式が`HH:MM`でない、などの
  パース不能・矛盾する行がないか
- 同一従業員番号・同一日付の行が重複していないか
- 列名が既知仕様と一致しているか（freee側の仕様変更で列が増減・改名されていないか）。
  一致しない場合はこの一致しない旨を`docs/freee_csv_notes.md`に記録すること

### 1. 打刻漏れ（過去1か月・前日以前）
`勤務日種別`が`所定労働日`にもかかわらず、`出勤時刻`または`退勤時刻`が空欄の日を
従業員ごとにリストアップ。休日・有休等の日は対象外。

**【重要・過去に発生したバグの修正】対象日付は「実行日より前」の日のみとし、
実行日当日・未来日は判定対象から明確に除外すること。** 過去に、今日・未来日
（まだ勤務していないため出退勤が空欄で当然の日）まで打刻漏れとして誤集計する
バグが発生した（本来0件のはずが90件超という異常値になっていた）。
`日付 < 実行日`という条件を必ず明示的にコードへ入れること（「過去1か月」という
期間の下限だけでなく、上限（実行日側）も正しく絞ること）。

**なお、直近日（実行日の1〜数日前）に打刻漏れが多く出ること自体は、
freee側のデータ反映ラグではなく、月末にまとめて打刻する等のルーズな実態を
正しく反映している可能性が高いことが判明している。そのため、直近日だからと
いって除外・猶予期間を設けたりせず、通常通りすべて集計・報告すること**
（是正を促す目的のため、むしろ検知すべき対象）。

**【重要・過去に発生した誤検知の修正】** `勤務日種別`が`所定労働日`であっても、
`勤怠種別`列が有休・欠勤・特別休暇等の「休暇系」の値になっている日は、
出勤・退勤時刻が空欄でも打刻漏れとして扱わないこと（休暇取得日は打刻不要が正常な
ため）。過去に`勤怠種別`列を見ずに`勤務日種別`だけで判定し、有休・欠勤日を大量に
誤検知したことがある（55名→本来22名まで是正された事例あり）。`勤怠種別`列の
実際の値を確認した上で、休暇系の値を網羅的に除外すること。

**【重要】該当者が多い場合でも「他n名」のように集約・省略しないこと。** 打刻漏れは
本人に個別に打刻を促す必要がある実務上の項目のため、**該当する全員の氏名を毎回
もれなく列挙する**こと（「月初のみだから軽微」等と判断して省略するのも禁止）。
人数が多くて長くなる場合でも、Slack投稿の該当項目に全員分を書き出すこと。

### 2. 22時以降の深夜勤務（直近1週間）
`深夜労働時間`が`00:00`以外、または`退勤時刻`が22:00より後（22:00ちょうどは除く）の日を
従業員ごとに検出。

### 3. 長時間労働の即時検知（直近1週間）
`総勤務時間`が10時間以上の日を検出。従業員ごとに「直近1週間で何日超過したか」を集計。

### 4. 週休2日の確保（直近1週間：月〜日）
`勤務日種別`が休日系（`所定休日`／`法定休日`等）の日数、または`総勤務時間`が
`00:00`かつ有休等で処理された日を休日とみなし、直近1週間の合計休日日数が2日未満の
従業員を検出。シフト勤務者も対象とし、曜日は問わず日数のみで判定する。

**【重要・シフト制従業員に関する注意】** `勤務・賃金`列が「シフト制」等の従業員は、
休日の記録方法が固定勤務者と異なり、現在のロジックでは正しく休日を検出できず
誤検知する可能性があることが判明している（過去の実行で実際に誤検知が発生した）。
シフト制の従業員がこの項目に該当した場合、**「該当」と断定せず、「要確認（シフト制の
ため自動判定が不正確な可能性あり）」という注記を付けて報告する**こと。
自動判定だけで緊急対応項目として扱わないこと。

### 5. 連続勤務のチェック
休日（上記4の定義）を取得せず6日以上連続勤務している従業員を検出。
シフト勤務者については、上記4と同様に「要確認（シフト制のため自動判定が不正確な
可能性あり）」の注記を付けて報告する。

### 6. 週次残業時間（直近1週間）
直近1週間の`時間外労働時間`を合計し、36協定の目安（週15時間）に近い・超えている
従業員を検出。目安の80%（週12時間）以上を「注意」、超過を「警告」として区別する。

### 7. 休日日数の総数（当月）
当月のカレンダー上の休日総数（週2日ペース×当月の週数で概算）と、各従業員の
実際の休日取得日数（上記4の定義の合計）を比較し、不足している従業員を検出。
シフト勤務者については、上記4と同様に「要確認（シフト制のため自動判定が不正確な
可能性あり）」の注記を付けて報告する。

### 8. 月次残業45時間アラート（当月）
当月の`時間外労働時間`合計が45時間を超えている、または経過日数から按分した
着地見込みが45時間を超えそうな従業員を検出。

### 9. 複数月平均80時間リスク（過労死ライン）
今回読み込んだ月のCSVの**`時間外労働時間`列を月間合計したもの**を月間残業時間として
`data/monthly_history.json`に追記・蓄積し（キーは従業員ID×年月。従業員IDは下記
「monthly_history.jsonのキー方式」を参照）、過去に蓄積された分と合わせて直近2〜6か月
（揃っている範囲）の平均を計算する。80時間超、または70時間以上で「注意」とする。
6か月分揃っていない場合は「データ収集中（現在n/6か月分）」と明記する。

**【重要・過去に発生したバグの修正】** 過労死ラインの残業集計には、必ず
`時間外労働時間`列を使うこと。**`法定内残業時間`列を誤って使ってはいけない**
（過去にこの列違いにより、残業時間が過小評価され、本来アラートすべき人を
見逃すリスクがあるバグが実際に発生した）。項目6・8と同じ列を使うことで一貫性を
保つこと。

#### monthly_history.jsonのキー方式
`data/monthly_history.json`の従業員キーは、**CSVの`従業員番号`を使うこと**
（旧freee API方式の`employee_id`は使わない）。過去に、CSV移行前の旧方式のキーと
CSV方式のキーが混在し、同一人物が別レコードに分裂するバグが発生し、
氏名で名寄せして統合する対応が必要になった。既存のファイルに旧方式のキーが
残っていないか実行のたびに確認し、もし残っていれば`従業員番号`ベースに統合してから
今回分を追記すること。

## Slack投稿（1回のみ）

### 投稿前の自己検証（必須。省略禁止）
投稿する直前に、以下を機械的に確認する。
- Slack本文に埋め込む「対象週」のラベル（MM/DD〜MM/DD）が、ステップ1で確定した
  `start_date`・`end_date`の変数の値と完全に一致しているか
- 実際に週次項目（2,3,4,5,6）の集計に使ったデータの日付範囲が、その`start_date`・
  `end_date`の範囲内に収まっているか（範囲外のデータを誤って含めていないか）
不一致があれば、投稿を中止し、`docs/freee_csv_notes.md`に不一致の内容を記録した上で、
代わりに「日付計算に不整合が見つかったため投稿を見送った」旨をSlackに1行だけ投稿する。

### 文字数制限への対応（Slackは1メッセージ最大4000文字程度が実用上の上限）
投稿する本文の文字数を事前にカウントし、**4999文字を超える場合は2回に分けて投稿する**
（Slack APIの上限に近づくと投稿が途中で切れる、または失敗するため）。
- 分割位置は、セクションの区切り（🚨緊急対応項目／🟡週次チェック項目／🔵月次・打刻
  チェック項目／要対応者サマリー）の間で行い、1文の途中で切らないこと。
- 目安として、1通目に「タイトル＋🚨緊急対応項目＋🟡週次チェック項目」、2通目に
  「🔵月次・打刻チェック項目＋要対応者サマリー」を割り振る。ただし、それでも
  いずれかの通が4999文字を超える場合は、さらに項目単位で分割する。
- 各メッセージの先頭に「（1/2）」「（2/2）」のように通し番号を付け、分割されている
  ことが分かるようにする。
- 分割した場合でも、Slack MCPへのリクエストは合計2回までに収め、3回以上には
  分割しないこと（それでも収まらない場合は、打刻漏れ等の該当者一覧を「上位10名＋
  他n名」のように圧縮してでも2通以内に収める）。

【表示ルール・最重要】
- 該当者がいない項目は「該当者：なし」の1行のみ。全員分の数値を列挙しない。
- 該当者がいる項目は、下記テンプレートの通り1行1名で列挙する。
- **例外：打刻漏れ（項目1）は、該当者が何名いても「他n名」のように省略せず、
  必ず全員の氏名を列挙すること。** 他の項目（3, 6等）は必要に応じて簡潔にまとめてよいが、
  打刻漏れだけは本人への個別対応が必要なため全員分を出す。
- シフト制従業員が項目4・5・7に該当する場合は「要確認（シフト制のため自動判定が
  不正確な可能性あり）」の注記を必ず添える。
- `data_quality_issues`がある場合のみ、投稿末尾に「⚠️ データ品質メモ」として簡潔に追記。
- 前月から見当たらなくなった従業員がいる場合のみ、投稿末尾に「ℹ️ 退職者確認メモ」として

### フォーマットの具体的なルール
- 各チェック項目は必ず**「見出し（太字）」→改行→「該当者：」→改行→1行1名の列挙**
  という縦の構造にする。項目名の直後に詳細を詰め込んだ1行にまとめないこと
  （可読性が悪いため、これが今回の修正理由）。
- 該当者が複数いる項目は、`・氏名：詳細（日付・時間等）`の形式で**1行に1名だけ**
  書く。複数名を読点やカンマで1行に並べて詰め込まないこと（打刻漏れの日付一覧を
  `氏名(日付,日付,日付)`のように1行に押し込むのも避け、日付が多い場合は
  `・氏名：日付1、日付2、日付3`程度に留め、読みやすい範囲を超えたら改行するなど、
  1名分のブロックが横に長くなりすぎないようにする）。
- 該当者がいない項目は「該当者：なし」の1行のみで、それ以上何も書かない
  （箇条書きの記号すら不要）。
- シフト制従業員が項目4・5・7に該当する場合は、該当者の行の末尾に
  「（要確認：シフト制のため自動判定が不正確な可能性あり）」を付ける。
- 大セクション（🚨／🟡／🔵）の区切りには`━━━━━━━━━━━━━━`の罫線を入れ、
  セクション間は必ず空行を1行以上入れる。
- 項目間（同じセクション内の項目と項目の間）にも空行を1行入れ、詰まった印象を
  避ける。
- 要対応者サマリーも同様に1行1名とし、`・氏名：該当項目（項目番号または項目名）`
  の形式で列挙する。

## 全体を通しての制約
- freee MCPへのリクエストは一切行わない（CSVのみを情報源とする）。
- Slack MCPへのリクエストは、CSV不足時のリマインドなら1回、分析完了後の結果投稿は
  文字数が4999文字以内なら1回、超える場合のみ最大2回まで（合計2回を超えない）。
- サブエージェントによる並列処理は使わない。
- 途中経過やデバッグ用の中間投稿はSlackに行わない。
