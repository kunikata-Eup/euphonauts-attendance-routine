## 2026-07-14: 「打刻漏れ（項目1）」が当月ファイルで系統的に過大検出される問題（未対策・要判断）

### 発生した問題
2026-07-14 8:00頃の実行で、対象期間（07/07〜07/13）を含む過去1か月の
「打刻漏れ」チェック（`勤務日種別=所定労働日`かつ`勤怠種別=未入力`かつ出退勤時刻空欄）を
実施したところ、**当月ファイル（`..._2026_08.csv`、実労働月2026年7月分）内で、
「本日に近い日付ほど未入力件数が跳ね上がる」という明確な勾配**が確認された。

`勤務日種別=所定労働日`の行のうち`勤怠種別=未入力`の件数を日付ごとに集計した結果：

| 日付 | 未入力件数 |
|---|---|
| 07/01〜07/03 | 11〜12件 |
| 07/06〜07/08 | 11〜14件 |
| 07/09〜07/10 | 17件 |
| **07/13（実行日前日）** | **40件** |
| 07/14（実行日当日） | 91件 |
| 07/15以降（未到来日） | 96〜100件 |

一方、**確定済みの前月ファイル（`..._2026_07.csv`、実労働月2026年6月分）**では、
月を通して1日あたり0〜5件程度で安定しており、月末（06/29・06/30）になっても
急増は見られなかった。

07/13時点で未入力だった40名のうち、6月にはまだ在籍していなかった新規従業員は
3名のみで、残り37名は6月から在籍している既存従業員だった（新規入社による
オンボーディング未整備が原因ではないことを確認済み）。

### 原因（推定）
freee側の実際の打刻・承認データが、CSVエクスポート時点では**直近の日付ほど
反映が追いついていない**可能性が高い（本人の打刻自体は完了していても、承認・
反映処理にタイムラグがあり、エクスポート時点でまだ「未入力」表示になっている
状態と推測される）。当月分（まだ締まっていない月）のファイルは、月初の日付
であっても、確定済みの前月ファイルと比べて未入力件数が有意に多い
（例：07/01の11件 vs 06/01の5件）。「本日に近いほど未入力が増える」という
明確な勾配があることから、単発の異常ではなく構造的な傾向と考えられる。

### 今回の対応
この状態のまま項目1を字面通り実行すると、**実際には打刻済み・処理中に過ぎない
可能性が高い日付を「打刻漏れ」として、実在する40名以上の従業員名とともに
Slackに投稿してしまうリスク**があると判断した。過去に発生した「勤怠種別を見ずに
55名を誤検知した」事例と同種の、実務上・心理的インパクトが大きい誤検知の
可能性があるため、**今回はSlackへの投稿を見送り**、この所見を本ファイルに記録した
上で、担当者（kunikata）にプッシュ通知で報告することにした。

### 今後の対策（要・人間による判断／Routine仕様への反映）
以下のいずれか（または組み合わせ）の対応を検討し、Routineプロンプトに反映することを
推奨する。
1. 項目1の集計対象から、**直近N日間（例：直近2〜3日）を除外**する
   （確定に一定のタイムラグがあることを前提にバッファを持たせる）。
2. 当月ファイルの`未入力`が、前月ファイルの同時期の水準と比べて明らかに
   高い場合は、`data_quality_issues`扱いとして注記に留め、個人名を伴う
   「打刻漏れ」としては扱わない、といった当月データの成熟度に応じた
   ロジックを追加する。
3. freee側の反映タイムラグの実態（何日で確定するか）を人間側で確認し、
   その日数に応じて上記1のNを決定する。

**このバッファ／閾値のロジックはRoutineの検出方針に関わる仕様判断のため、
Claude側で独断で追加せず、次回実行前に担当者の指示を確認すること。**
