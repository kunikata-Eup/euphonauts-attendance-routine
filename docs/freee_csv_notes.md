# freee CSV運用メモ・学習ログ（勤怠コンプライアンスチェックRoutine用）

このファイルは、routine_spec.md の【最重要・厳守事項】6.に基づき、実行中に判明した
想定外の事象・仕様・対処方針を蓄積するファイルです。次回実行時は必ず先にこのファイルを
読んでから作業を始め、同じ調査を繰り返さないようにしてください。

## 2026-07-15 実行分

### 1. `時間外労働時間`列は、進行中（未確定）の当月分CSVでは全日0のまま出力される

- 確定済み月（6月分CSV, `..._2026_07.csv`）では`時間外労働時間`に実データが入っている
  （2730行中83行が非ゼロ）。
- 一方、進行中の当月分CSV（7月分, `..._2026_08.csv`）は、実行日時点で**全3,100行が
  例外なく`00:00`**だった（`所定内労働時間`や`総勤務時間`はリアルタイムで正しく入っている
  ことは確認済み）。フレックスタイム制の精算方式上、残業時間は月次確定処理後にまとめて
  計算・反映される仕様と推測される。
- **影響**：項目6（週次残業）・項目8（月次45時間アラート）・項目9（複数月平均80時間リスク）
  は、`時間外労働時間`列に依存するため、月の途中では実質的に「該当者なし」以外の結果が
  出ない状態になる。これはチェック漏れというより「データ未確定」に起因するものだが、
  Slack投稿では必ず「⚠️ データ品質メモ」としてこの旨を明記し、油断されないようにすること。
- 次回以降、この仕様が変わっていないか（＝当月分が引き続き全0か）を確認し、もし
  途中から値が入るようになっていた場合はこの記述を更新すること。

### 2. `data/monthly_history.json` を旧freee API方式からCSV方式へリセットした

- 既存ファイルは2026-07-04時点（CSVパイプライン移行前）の調査時に作成された、旧freee API
  方式のテストデータだった：キーが`employee_id`（例："1943218"）、フィールド名が
  `total_excess_statutory_work_mins`等でCSVの列と対応が不明瞭、対象も5名分のみ、
  かつ2026-07分は月初数日分の途中経過値だった。
- routine_spec.mdの指示（従業員番号キー・`時間外労働時間`ベースで統一すること）に
  照らして非互換と判断し、労務リスク判定（過労死ライン等）に不確かなデータを混入させる
  リスクを避けるため、**そのまま流用せず、CSV方式（`従業員番号`キー、
  `{"name":..., "months": {"YYYY-MM": {"overtime_minutes": 分}}}`構造）でリセットした**。
- そのため今回のSlack投稿では項目9は全員「データ収集中（現在1/6か月分）」扱いとした。
  次回以降、このファイルに月次データが正しく蓄積されているか確認すること。

### 4. シフト制従業員は項目1（打刻漏れ）でも自動判定が不正確になりうる

routine_spec.mdは項目4・5・7についてのみ「シフト制の場合は要確認」の注記ルールを定めているが、
実際に調査した結果、**項目1（打刻漏れ）でも同じ問題が起きる**ことが判明した。

- 該当例：大橋海（勤務・賃金＝シフト制）。実働日（17:30〜翌9:30の隔日夜勤パターン）以外の日は
  `勤務日種別`が`所定労働日`のまま、`勤怠種別`が`未入力`になっている。
- `勤務予定開始時刻`/`勤務予定退勤時刻`列も確認したが、全日一律`00:00`〜`23:59`で、
  実働予定日か否かの判別には使えなかった。
- そのため月の大半（今回は27/31日）が機械的に「打刻漏れ」として検出されてしまう。
  これは実際の打刻忘れというより、シフト制特有の勤務日種別の記録方式に起因する誤検知の
  可能性が高いと判断し、今回は項目1でもシフト制従業員には項目4/5/7と同様の
  「（要確認：シフト制のため自動判定が不正確な可能性あり）」の注記を付けて報告した
  （列挙自体は省略していない）。
- 次回以降も同じ扱いを踏襲してよい。より正確な判定方法（例：実働日の出退勤パターンから
  シフトの実働サイクルを推定する等）が判明したら、この項を更新すること。

### 5. 項目5（連続勤務）の対象データ範囲について

投稿前の自己検証ルール上は項目5も「週次項目」として`start_date`〜`end_date`のデータのみで
判定すべきと読めるが、文字通り対象週の7日分のみに限定すると、**週をまたぐ連続勤務
（例：先週後半から今週前半にかけての7連勤）を見逃す**ことが実際に判明した
（千葉峻生：07/05〜07/11の7連勤。対象週(07/08〜07/14)のデータだけで判定すると
最大連続日数は4日にしかならず、検出できなかった）。

これは「対象週ラベルと集計対象のズレ」を防ぐための自己検証ルールの趣旨（コードのバグで
無関係な期間を混入させない）とは別種の問題であり、6日以上連続勤務という指標自体が
本質的に週境界をまたぐ性質を持つため、**項目5に限り、当月に読み込んだ全期間のデータで
連続日数を判定し、該当者にはSlack投稿で実際の連続期間（開始日〜終了日）を明記する**方式とした
（ラベルとのズレが生じないよう、日付範囲を明示することで担保）。次回以降もこの方式を踏襲する。

### 6. move_csv.bat側のログに「push: 成功」と表示されても実際には失敗している場合がある

`data/manual_csv/move_log.txt`を確認したところ、2026/07/15の複数の実行ログで
`! [rejected] HEAD -> main (fetch first)`というgit pushの実際の失敗メッセージの直後に
`push: 成功（push先: main）`という矛盾した行が出力されていた（バッチスクリプト側が
git pushの終了コードを見ずに常に成功と表示している可能性がある）。

ただし今回はステップ0-a(a)の判定基準通り`move`コマンドの生出力で対象ファイルの移動自体は
確認できており、かつ最終的にはリモート側で自動マージ（`Merge branch 'main' of ...`）が
成立し、本Routineが参照するorigin/main上には当日分のデータが反映されていたため、
本Routineの実行自体への支障はなかった。ただし、これは移動処理と鮮度判定がgit pushの
成否に依存しない設計（move_log.txtのmoveコマンド出力ベース）になっていたための
結果オーライであり、move_csv.bat側のpush失敗検知ロジック自体は改善の余地があると考えられる。
労務担当者側で一度確認することを推奨（Routine側での対応は不要）。

### 7. 6月は通常勤務だった3名が7月に入って完全に打刻が途絶えている

市野瑞穂・高野涼輔・三浦風薫（三浦は07/03以降）の3名は、6月分CSVでは`通常勤務`が
20件以上あったのに対し、7月分CSVでは対象月初から一度も`通常勤務`の打刻がなく、
`未入力`が連続している。除外リスト（対象外4名・育休2名）には含まれていない。
今回は項目1の該当者として通常通り列挙したが、単なる打刻忘れの範囲を超えている
可能性があるため、労務担当者側での個別確認を推奨する。
