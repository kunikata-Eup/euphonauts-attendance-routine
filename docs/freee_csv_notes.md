# freee CSV 学習ログ

このファイルは、勤怠コンプライアンスチェックRoutine（`docs/routine_spec.md`）の実行中に
発生した想定外の事象・原因・対策を記録する学習ログです。次回実行時は必ずこのファイルを
読んでから作業を始めること。

---

## 2026-07-14 実行分

### 1. 進行中の月は「時間外労働時間」「法定内残業時間」列が常に00:00になる

**発生した問題：** 2026年7月分CSV（`..._2026_08.csv`、実労働月として進行中）で、
`時間外労働時間`列・`法定内残業時間`列が全3131行とも`00:00`だった。一方、`総勤務時間`
`深夜労働時間`等の実績列は日々正しく記録されていた。
例：大塚純也 2026/07/09 出勤09:00-退勤21:00（総勤務時間=10:00）にもかかわらず
`時間外労働時間`=`00:00`・`法定内残業時間`=`00:00`。

締め済みの2026年6月分CSV（`..._2026_07.csv`）では、同列に実値が入っていることを確認済み
（例：田中大地 6月合計306分）。

**原因（推測）：** freee側で残業時間の計算は月末締め処理後に確定する仕様と思われる。
進行中の月はまだ計算されておらず、便宜上00:00で出力されている可能性が高い。

**今後の対策：**
- 進行中の月については、項目6（週次残業）・項目8（月次45時間アラート）を
  「算出不可」として報告し、「なし（=残業ゼロを確認済み）」と混同しないこと。
- 項目9（複数月平均80時間リスク）の平均計算からは、進行中の月（当月）を除外し、
  締め済み月のみを使うこと（そうしないと当月分が常に0扱いとなり、平均が
  不当に押し下げられ、本来アラートすべき人を見逃すリスクがある）。
- 月末〜翌月頭に再実行された際、`時間外労働時間`列に実値が入るようになるか
  次回以降も継続して確認すること。もし月末を過ぎても00:00のままの場合は、
  この仮説が誤っている可能性があるため、freee側の設定を要確認。

### 2. monthly_history.json に旧方式（freee API employee_id）キーが残存していた

**発生した問題：** `data/monthly_history.json`に、CSV移行前の旧方式キー
（例：`1943218`, `1943221`, `1943222`, `1986176`, `1986181`）が5名分残っており、
現在のCSVの`従業員番号`（それぞれ 2, 3, 4, 7, 8）と一致していなかった。
仕様書が警告している「同一人物が別レコードに分裂するバグ」の実例が既に発生していた状態。

**対策実施内容：** 氏名（空白除去・正規化）で名寄せし、該当5名分を
CSVの`従業員番号`ベースのキーに統合して`data/monthly_history.json`を更新・コミットした。

**今後の対策：** 実行のたびに、`monthly_history.json`のキーが現在のCSVの`従業員番号`一覧に
存在するか必ず確認し、存在しない（＝旧方式または退職者の）キーが見つかった場合は
氏名で名寄せを試みること。

### 3. monthly_history.json の残業時間が「法定内残業時間」列から誤って算出されていた

**発生した問題：** 上記5名分の2026年6月データについて、記録されていた残業時間
（旧フィールド名`total_excess_statutory_work_mins`）が、実際には`法定内残業時間`列の
月合計から算出されていたことが判明した（例：小川陽暉 316分＝`法定内残業時間`列合計と一致、
`時間外労働時間`列合計は0分）。仕様書が明記する「過去に発生したバグ」と同種の実例。

**対策実施内容：** 該当5名×6月分について、`時間外労働時間`列から再計算し、
フィールド名も`total_overtime_mins`に変更して修正した。

**今後の対策：** `monthly_history.json`を書き込む処理は必ず`時間外労働時間`列のみを
参照すること。`total_overtime_mins`というフィールド名を維持し、
`total_excess_statutory_work_mins`という紛らわしい旧フィールド名は今後使わないこと。

### 4. 項目7（休日日数不足・当月）の按分について

仕様書には項目7の「経過日数按分」についての明記がなかったが、月半ばに満月分の
休日目安（週2日×当月週数）とそのまま比較すると、ほぼ全従業員が「不足」と誤検知される
リスクが高いと判断した。項目8（月次45時間アラート）が「経過日数から按分した着地見込み」
を明示的に採用していることと整合させ、項目7でも当月の経過日数（実行日時点で経過した週数
×2日）を目安として比較する方式を採用した。次回以降もこの方式を踏襲すること。
