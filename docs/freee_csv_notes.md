# 勤怠CSVチェックRoutine 学習ログ（freee CSV方式）

このファイルは、CSVエクスポート方式（freee MCP/APIを呼ばずリポジトリ内CSVのみで
分析する運用）における「発生した問題」「原因」「今後の対策」を蓄積する学習ログ。
**次回実行時は必ずこのファイルを最初に読んでから作業を始めること。**

---

## 2026-07-14 実行分

### 発生した問題

今回のRoutine起動時に渡されたタスク説明文が、「ステップ0：パイプライン起因の異常を
検知する軽量な事前検証」のうち「0-x) 作業ブランチの確認」の手順3
（「1と2が一致しない場合、以下を実行してデフォルトブランチに切り替える。」）の直後で
途切れており、それ以降の内容が本セッションに渡されなかった。具体的に欠落していたのは：

- 0-x手順3で実行すべき実際のgitコマンド
- 事前検証 a〜e の各異常パターンの定義（本文中で「a〜eのいずれかで異常が見つかった
  場合」と言及されているが、a〜eの中身自体は一度も提示されなかった）
- ステップ1「対象期間の確定」のロジック
- 9項目チェックそのものの定義内容（列名の仕様は詳細に提示されたが、「10時間以上を
  長時間労働とする」以外の具体的な判定基準・閾値・9項目の内訳は提示されなかった）
- 「CSVが無い場合の処理」の具体的な手順
- Slack投稿のフォーマット・投稿先チャンネルID（チャンネル名 `#hr_勤怠アラート_自動通知`
  のみ判明）

### 原因

不明。プロンプトがセッションに渡される過程で末尾が切り詰められた可能性が高いと
推測されるが、断定はできない。過去の `docs/freee_api_notes.md`（旧API方式時代のログ）
にはこの種の「指示文の途中欠落」の記録はなく、今回が初めて確認された事象。

### 今回の対応（安全側に倒した判断）

1. **9項目チェックの分析・Slack投稿は実施しなかった。** チェック項目の正式な定義が
   与えられていない状態で独自に9項目を推測・創作して業務用Slackチャンネル
   （`#hr_勤怠アラート_自動通知`）に「労務コンプライアンス分析結果」として投稿するのは、
   HR判断を誤らせるリスクがあるため行わないと判断した。
2. 代わりに、リポジトリ所有者（kunikata@euphonauts.co.jp）へ直接通知し、次回実行時に
   完全な指示文が渡されるよう確認を依頼した。
3. 「0-x 作業ブランチの確認」自体は、判明していた範囲で実施済み：
   - `git remote show origin` → HEAD branch は `main`
   - `git branch --show-current` → 本セッションは `claude/fervent-pascal-k30brd`
   - 両者のブランチ名は不一致だが、`git fetch origin main` 後に比較したところ、
     本セッションの HEAD コミット（`9602709`）は `origin/main` の HEAD コミットと
     **完全に一致**していた。つまりデータの鮮度には問題がないと判断した。
   - **ブランチの切り替えは行わなかった。** Claude Code on the web のセッションは
     常に専用の `claude/*` 作業ブランチ上で動作する制約があり、他のブランチへの
     切り替えやそれ以外のブランチへのpushは方針上禁止されているため。今後も
     同様の状況では「切り替え」ではなく「`git fetch` + コミットハッシュ比較」で
     鮮度検証を代替する運用とする。

### 今後の対策

1. 次回実行時、タスク説明文に9項目チェックの定義・ステップ1以降の手順・
   Slack投稿フォーマット・投稿先チャンネルIDが正しく含まれているか、作業開始前に
   必ず確認すること。欠落していた場合は今回と同様に分析・投稿を行わず、安全側で
   停止してユーザーに直接通知すること。
2. 9項目チェックの正式な定義や「CSVが無い場合の処理」等の恒久的な仕様は、
   Routineの起動プロンプト（毎回渡される可変的なテキスト）だけでなく、
   このリポジトリ内のドキュメント（例：本ファイルや専用の仕様ファイル）にも
   保存しておくことをユーザーに提案する。プロンプトのみに存在する仕様は、
   今回のような伝搬事故（途中欠落）に弱い。
