# freee CSV運用ノート（学習ログ）

このファイルは `docs/routine_spec.md` のルール6に基づき、実行中に発生した想定外事象・
発見事項・今後の対策を蓄積する。次回実行時は必ず本ファイルを先に読むこと。

## 2026-07-14 実行分

### 1. move_log.txt はcp932単一エンコーディングではない（混在ファイル）
`data/manual_csv/move_log.txt` をルール通り`cp932`で全体デコードしようとしたところ、
`UnicodeDecodeError`（位置461付近）が発生した。調査の結果、このログファイルは
**cp932とUTF-8が混在**していることが判明した。
- Windowsの`move`コマンド／バッチ側の出力（ファイル名・「◯個のファイルを移動しました」等）
  → **cp932（Shift_JIS）**
- 後段でバッチから叩いている`git`コマンドの出力（`warning: ... LF will be replaced by CRLF`、
  一部の`commit:`/`push:`ステータス行など）→ **UTF-8**
同一ファイル内でこの2つが混在するため、単純な「ファイル全体をcp932でデコード」では
必ずどこかで失敗する。

**今回の対策・今後の推奨**：1行ずつ読み込み、`cp932`でデコードを試みて失敗したら
`utf-8`にフォールバックする方式で全行を復元した（フォールバックも失敗する行は
発生しなかった）。次回以降もこの「行単位でcp932→utf-8フォールバック」方式を
使うこと。`routine_spec.md`のステップ0-a「cp932でデコードして読み込む」という
記述は不正確なので、次回spec更新時に修正を検討してよい。

### 2. 当月分（未締め）CSVは「時間外労働時間」列が全行00:00になる
今回読み込んだ実労働月の当月分ファイル（`..._2026_08.csv`, 実労働月2026年7月分）は、
**全3131行にわたって`時間外労働時間`列・`法定内残業時間`列が"00:00"で統一**されていた
（`法定休日労働時間`にわずかに非ゼロあり）。一方、確定済みの前月分ファイル
（`..._2026_07.csv`, 実労働月2026年6月分）では同列に正常な非ゼロ値が多数入っていた。

対象従業員の大半が`勤務・賃金`=`フレックスタイム`であることから、freee側の
仕様として**フレックスタイム勤務者の残業時間は月次締め処理が走るまで算出されず、
月の途中は一律0として出力される**可能性が高いと推測される（パース失敗や
列違いのバグではなく、データ自体がそうなっている）。

**影響**：この特性により、項目6（週次残業）・項目8（月次残業45時間、当月分）は、
月の前半〜中盤は構造的に「該当者なし」になりやすい。これはバグではなく、
freee側のデータ確定タイミングに起因する制約である可能性が高い旨、今回のSlack投稿では
触れていないが、次回以降もこの現象が続く場合は運用者への注記を検討すること。
項目9（複数月平均80時間）はmonthly_history.jsonへの月次確定値の蓄積により、
月末に確定した値が翌月以降の平均計算に反映されるため、長期的なリスク検知は
機能する設計になっている。

### 3. monthly_history.json に旧方式（employee_id）キーが残存していた（今回移行済み）
実行開始時点の`data/monthly_history.json`には、5名分について**旧freee API方式の
employee_id**（例: `1943218`）をキーとするレコードが残っていた。CSVの`従業員番号`列
（例: 小川陽暉は`2`）と照合し、氏名一致により以下の通り**従業員番号ベースのキーに
統合済み**：
- `1943218` → `2`（小川陽暉）
- `1943221` → `3`（田中大地）
- `1943222` → `4`（野口智哉）
- `1986176` → `7`（髙野悠星）
- `1986181` → `8`（紺野将人）

今回のコミットで反映済みのため、次回実行時点では旧方式キーは残っていないはずだが、
念のため次回実行時も「氏名から逆引きしたCSVの従業員番号とキーが一致しているか」の
確認は継続すること。
