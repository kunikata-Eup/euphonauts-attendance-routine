# freee CSV（日次勤怠一覧）学習ログ

このファイルは、CSV手動アップロード方式の勤怠コンプライアンスチェックRoutineが、
実行のたびに発生した想定外の事象・仕様の詳細を記録していく学習ログです。
次回実行時は必ずこのファイルを読んでから作業を始めること。

## 初回フル実行（2026-07-10）で判明した事項

### 1. 実際のCSV列は既知仕様（20列）より多い（52列）
`data/manual_csv/日次勤怠一覧_月末締め（翌月末日払い）_2026_08.csv` のヘッダーを確認したところ、
既知仕様に記載の20列（氏名・従業員番号・部門・日付・曜日・勤務日種別・勤怠種別・出勤時刻・
退勤時刻・所定内労働時間・法定内残業時間・時間外労働時間・法定休日労働時間・深夜労働時間・
遅刻時間・早退時間・休憩時間・総勤務時間・有休時間 - 合計・勤怠メモ）は**すべて列名が完全一致**
で存在していたが、それ以外に以下の列が追加で存在した（合計52列）。
`申請アラート`, `勤務・賃金`, `勤務予定開始時刻`, `勤務予定退勤時刻`, `遅刻早退時間`,
`控除対象の遅刻早退`, `有休時間 - 半休`, `有休時間 - 時間休`, `有給特休時間 - 合計/半休/時間休`,
`無給特休時間 - 合計/半休/時間休`, `始業時刻1〜3`, `終業時刻1〜3`, `休憩1〜3入り/戻り時刻`,
`出勤登録種別`, `退勤登録種別`, `出勤/退勤打刻変更前/後時刻`。
→ 既知仕様の20列が存在していれば問題なく処理できるため、これはブロッカーではない。
`勤務・賃金`列（値は`シフト制`／`フレックスタイム`）はシフト制判定に実際に使用している。
今後freee側の列がさらに変わっても、既知20列さえ維持されていれば処理継続可能。

### 2. `勤怠種別`列の実際の値（休暇系判定に使用）
観測された値：`所定休日`, `法定休日`, `有休`, `有休（半休）`, `欠勤`, `未入力`, `通常勤務`,
`趣味休暇`, `趣味休暇（半休）（半休）`。
→ 打刻漏れ（項目1）の判定では、`未入力`と`通常勤務`以外はすべて「休暇系」として除外している。
→ `趣味休暇（半休）（半休）`は「（半休）」が二重についているが、freee側のカスタム休暇種別の
表示仕様とみられる（データ品質の異常ではなく、そのままの値として扱えばよい）。

### 3. 有休取得日の`総勤務時間`は`00:00`にならない
仕様書には「`総勤務時間`が`00:00`かつ有休等で処理された日を休日とみなし」とあるが、実際の
`有休`（終日）の行は`総勤務時間`に所定労働時間相当（例：`08:00`）が入り、`00:00`にはならない
（`出勤時刻`・`退勤時刻`は空欄）。そのため週休2日判定（項目4等）では、この条件は現状ほぼ
発火せず、実質的に`勤務日種別`が`所定休日`／`法定休日`の日のみを休日としてカウントしている。
これは労務的にも正しい（有休は所定労働日の欠勤扱いであり、法定・所定の休日にはならない）ため、
現状のロジックのままでよいと判断した。ただし将来、`総勤務時間=00:00`かつ有休処理される
ケース（例：無給特別休暇等）が実際に観測されたら、その時点でこのメモを更新すること。

### 4. `move_log.txt`は「直近の実行の commit/push 確認行」が常に欠けて見える
`move_log.txt`は「move結果を記録→git add/commit→（committされた内容には含まれない）
commit/push結果を追記→次回実行時のgit addでようやくその追記分がコミットされる」という順序で
書かれているらしく、**その日の最新エントリだけを見ると、必ずmove結果のみで終わっており、
commit/push確認行が欠けているように見える**（実際にはcommit/pushは成功しており、次回実行時に
その確認行が追記されてコミットされる）。
→ 鮮度チェック（ステップ0-a）では、moveコマンドの生出力に対象ファイル名が含まれているかだけを
見れば十分で、直近エントリにcommit/push確認行が無いこと自体は異常ではない。誤診断しないこと。

### 5. `data/monthly_history.json`に旧freee API方式のキーが残っていた
2026-07-10実行時点で、`data/monthly_history.json`には旧フェーズ（freee API直接取得方式）で
使われていた`employee_id`（例：`1943218`）がキーとして残っていた（氏名は`小川陽暉`のような
スペース無し表記）。CSV方式の`従業員番号`（例：`00002`）とは全く異なる値だったため、
氏名（空白除去して比較）でCSVの`従業員番号`と突き合わせてキーを移行した。
旧エントリの残業時間フィールド名は`total_excess_statutory_work_mins`（freee API
work_record_summariesのフィールド）だったが、フィールド名の意味（「法定労働時間を超過した分」）
から、CSVの`時間外労働時間`と同じ概念（法定外残業）を指すと判断し、そのまま実働月数の
カウントに含めた（`法定内残業時間`とは別物と判断）。今後CSVベースの`overtime_mins`
フィールドで上書きされていくため、実害はない。
→ 今回の移行で対象になった従業員：小川陽暉→00002、田中大地→00003、野口智哉→00004、
髙野悠星→00007、紺野将人→00008（他、氏名一致した全員）。次回以降の実行では、この移行は
既に完了しているはずなので、`data/monthly_history.json`のキーが従業員番号形式になっている
ことを確認するだけでよい。

### 6. シフト制従業員（大橋海）は`勤務日種別`が常に`所定労働日`
シフト制（`勤務・賃金`=`シフト制`）の大橋海さんは、2026年6月・7月の全期間を通じて
`勤務日種別`が一度も`所定休日`／`法定休日`になっていない（実際に出勤していない日も
`所定労働日`のまま、`勤怠種別`は`未入力`）。そのため、項目4・5・7の自動判定ロジックでは
休日を正しく検出できず、機械的には「休日取得ゼロ」「連続勤務39日」等の極端な値になる。
これはシフト制従業員に既知の自動判定不正確問題そのものであり、Slack投稿では
「要確認（シフト制のため自動判定が不正確な可能性あり）」の注記を付けて報告している
（緊急対応項目として断定しない）。freee側のシフト制従業員の休日記録方法について、
人事側に一度確認することを推奨する。
