# freee CSV方式 勤怠コンプライアンスチェック 学習ログ

CSV手動アップロード方式（`data/manual_csv/`）で発生した想定外の事象と対策を記録する。
次回実行時は必ずこのファイルを読んでから作業を始めること。

## 2026-07-11 初回フル実行時の発見事項

### 1. move_log.txt は純粋なCP932ではない（混在エンコーディング）

`data/manual_csv/move_log.txt` を `cp932` で一括デコードすると
`UnicodeDecodeError: 'cp932' codec can't decode byte 0x81 in position ...` で失敗する。

**原因**：このログファイルは `move_csv.bat` 自体（Windowsコンソール／`move`コマンドの出力）は
CP932で書き込まれる一方、バッチ内から呼び出す `git` コマンドの標準エラー出力
（例：`warning: in the working copy of '...', LF will be replaced by CRLF...` や
コミットハッシュ行）はUTF-8で出力されるため、**同一ファイル内にCP932とUTF-8が行単位で
混在している**。

**対策**：ファイル全体を一括で `cp932` デコードせず、**行ごとに「まずUTF-8として
デコードを試み、失敗したらCP932にフォールバックする」**方式でデコードすること。

```python
def decode_lenient(line_bytes):
    try:
        return line_bytes.decode('utf-8')
    except UnicodeDecodeError:
        return line_bytes.decode('cp932', errors='replace')

lines = raw_bytes.split(b'\n')
text = '\n'.join(decode_lenient(l) for l in lines)
```

### 2. シフト制従業員は「打刻漏れ（項目1）」も誤検知しうる

既知の注意点（項目4・5・7）は「シフト制従業員は休日の判定方法が固定勤務者と異なり、
`勤務日種別`だけでは休日を正しく検出できない」というものだったが、今回の実行で
**項目1（打刻漏れ）も同じ理由で誤検知することが判明した**。

シフト制従業員（例：大橋海）は、シフトが入っていない日でも `勤務日種別=所定労働日`
のまま `勤怠種別=未入力`・出退勤時刻空欄という記録になる（固定勤務者のように
`所定休日`／`法定休日`にはならない）。そのため、単純に「所定労働日で出退勤時刻が
空欄」を打刻漏れと判定するロジックでは、シフトが入っていないだけの日を大量に
誤検知してしまう（今回の実行では対象期間中の約8割の日が「打刻漏れ」として
検出された）。

**対策**：項目1の集計・Slack投稿時、`勤務・賃金=シフト制`の従業員については
「要確認（シフト制のため休日と勤務日の自動判定が不正確な可能性あり）」の注記を
付けて報告する（項目4・5・7と同様の扱いに準ずる）。全員列挙のルールは維持しつつ、
注記で判断材料を添える運用とした。

### 3. 深夜労働時間・退勤時刻が「24:00」を超える表記になることがある

シフト制の夜勤（例：出勤17:30〜退勤33:30）では、退勤時刻が日をまたいで
「33:30」（＝翌9:30）のように24時間を超える形式で記録される。これは異常値ではなく、
日本の勤怠システムでよく使われる「日またぎ勤務の翌日時刻を24以上の値で表す」表記。
一見異常に見えるが、`深夜労働時間`列の値（今回は6時間程度）と整合していれば
正常な夜勤記録である可能性が高い。パース時は `HH:MM` の `HH` が24以上でも
エラーとせず、`HH*60+MM` 分として扱ってよい。ただし通常勤務者でこの表記が
出た場合はデータ異常の可能性があるため、シフト制以外の従業員で発生した場合は
`data_quality_issues` に記録すること。

### 4. 対象ブランチについて

このRoutineのクローンが参照していたブランチ（`claude/brave-lovelace-jlede8`）は、
実行時点で既にリモートから削除されていた（マージ済みのPRのブランチだったと
推測される）。デフォルトブランチ（`main`）とはコミットハッシュが完全一致していたため
分岐はなく、`main`から同名でブランチを作り直して作業を継続した。CSVの実体は
既に`main`に反映済みだったため、データの欠落や不整合は発生していない。
