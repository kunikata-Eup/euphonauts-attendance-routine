# freee CSV方式 運用メモ（勤怠コンプライアンスチェック用）

このファイルはCSV方式（`data/manual_csv/`配下の手動アップロードCSVを読み込む方式）の
学習ログ。次回実行時は必ず読んでから作業を始めること。

## 初回フル実行（2026-07-07）で判明した事項

### 1. 列名は「既知仕様」より多い（異常ではない）
実際のCSVには、既知仕様に記載した主要列に加えて以下のような列が存在する：
`申請アラート`, `勤務・賃金`, `勤務予定開始時刻`, `勤務予定退勤時刻`, `遅刻早退時間`,
`控除対象の遅刻早退`, `有休時間/有給特休時間/無給特休時間`の内訳列, `始業時刻1-3`,
`休憩1-3入り/戻り時刻`, `出勤登録種別`, `退勤登録種別`, 打刻変更前後時刻など。
既知仕様の主要列は全て存在しており、欠落はない。列が増えていること自体は
フォーマット異常ではないため、今後も「既知の主要列が全て存在するか」だけを
チェックすればよい（列数の完全一致は求めない）。

### 2. シフト制従業員は「勤務日種別」が実態を反映していない（重要な新規判明事項）
現在シフト制の従業員は社内に1名（大橋海／従業員番号00088）。この従業員のCSVを見ると、
**実際には勤務日でない日（休みの日）も含めて全ての日の`勤務日種別`が「所定労働日」に
なっている**（休みの日でも`所定休日`／`法定休日`にならない）。また`総勤務時間`が
`00:00`でも`勤怠種別`が`未入力`のままになっており、休日なのか単なる打刻漏れなのか
`勤務日種別`だけでは区別できない。

この結果、`勤務日種別`に依存する以下のロジックがシフト制従業員に対して正しく機能しない：
- **項目1（打刻漏れ）**：休みの日が「所定労働日なのに未入力」＝打刻漏れとして
  大量に誤検知される（今回の実行で大橋海に23件の偽陽性が発生した）
- **項目4（週休2日）**：休日が全く検出されず「休日0日」と誤判定される
- **項目5（連続勤務）**：休日が検出されないため、連続勤務日数が実質「データ期間全体」
  まで伸び続けてしまう（今回36日という非現実的な値が出た）
- **項目7（休日日数不足）**：項目4と同様の理由で誤判定される

**対応方針（今回適用・次回以降も踏襲すること）**：
`勤務・賃金`列が`シフト制`の従業員は、項目1・4・5・7の自動判定結果をそのまま
「該当」として報告せず、「要確認（シフト制のため自動判定が不正確な可能性あり）」と
明記する。項目1については、該当者リストから除外した上で、この注記を別途1行添える
（大量の偽陽性の日付を列挙するとかえって誤解を招くため）。
項目2（深夜勤務）・項目3（長時間労働）は`勤務日種別`に依存せず実際の打刻・実働時間
（`退勤時刻`・`深夜労働時間`・`総勤務時間`）を直接見ているため、シフト制でも
正しく判定できる。この2項目は通常通り報告してよい。

もしシフト制従業員が増えた場合も、同じロジックで`勤務・賃金`列を見て機械的に
除外・注記すればよい（従業員名のハードコードではなく`勤務・賃金=='シフト制'`の
条件で判定していること）。

### 3. monthly_history.jsonのキー移行を実施
初回CSV実行時点で、`data/monthly_history.json`に旧freee API方式時代の
`employee_id`キー（例：`1943218`）が5件残っていた。氏名で名寄せし、CSVの
`従業員番号`（例：`00002`）をキーとする形式に移行済み。今後はこのキー形式で
統一する。移行対応表（記録として残す）：
- 1943218 → 00002（小川陽暉）
- 1943221 → 00003（田中大地）
- 1943222 → 00004（野口智哉）
- 1986176 → 00007（髙野悠星）
- 1986181 → 00008（紺野将人）

またmonthly_history.jsonのスキーマも、CSV方式に合わせて簡素化した
（`overtime_mins`＝その月の`時間外労働時間`列合計・分、のみを保持する形に変更）。
過去のAPI方式時代の`total_work_mins`等のフィールドは今後使用しない。

### 4. 項目7（当月休日日数不足）の按分方法についての判断
実行日が月初に近い場合、「当月の週数×2日」を丸ごと期待値にすると、月初は
ほぼ全員が「不足」と誤検知されてしまう。そのため、**当月の経過日数ベースで
期待値を按分する**（経過週数×2日）方式を採用した。例：7/7実行時点で7月の
経過日数は6日（7/1〜7/6）→期待値は約1.7日。この按分方式を今後も踏襲すること。

### 5. move_log.txtの判定は「本日の全エントリ」を確認すること
本日1日で`move_csv.bat`は複数回実行される（実際に7/7だけで6回実行された）。
そのうち実際にファイルを移動できたのは2回のみで、残りは
「重複するファイル名が存在するか、またはファイルが見つかりませんでした」
（＝Downloadsフォルダに新規ファイルがなかっただけ）だった。**「ログの最新
エントリ」1件だけを見て判定すると、たまたま最後の実行が「新規ファイルなし」の
回だった場合に誤って異常と判定してしまう。** 本日分の全エントリを走査し、
その中に対象支給月ファイル名を含む移動成功記録が1件でもあれば正常と判定すること。

### 6. move_csv.bat側のスクリプトバグ（人間側への申し送り事項）
2026/07/07 12:13の実行時、move_log.txtに「Default branch: branch」「push先: branch」
という記録があり、pushが本来のブランチ名ではなく文字列`branch`という名前の
ブランチに対して行われていた形跡がある。ローカルスクリプト側でブランチ名の
取得に失敗している可能性があるため、人間側での`move_csv.bat`の確認を推奨する
（Routine側では対応不可のため次回実行時に気づいた場合はSlack投稿末尾等で
一言申し送りすること）。

### 7. 作業ブランチについて
このRoutineの実行環境では、`git remote show origin`のHEAD branchが
実行ごとに変わるランダムな名前（例：`claude/sharp-sagan-nox10q`）になっており、
実質的には各回のRoutine実行が使う一時ブランチを指しているだけで、恒久的な
「default branch」を意味しない。今回確認したところ、そのブランチと実際の
作業ブランチのコミット内容（HEAD SHA）は完全に一致しており、実害はなかった。
今後もこのチェックでブランチ名の不一致が出た場合は、まずコミット内容
（`git log`のSHA比較）が一致しているかを確認し、内容が同一であれば
無理な切り替えは行わず、その旨だけ記録すればよい。
