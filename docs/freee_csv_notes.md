# freee CSV方式 学習ログ

CSV手動アップロード方式の勤怠コンプライアンスチェックRoutineを実行する際に判明した、
仕様書（Routineプロンプト）記載の「既知仕様」とは異なる挙動や、判断に迷った点を
実行日順に記録する。次回実行時は必ずこのファイルを読んでから作業を始めること。

## 2026-07-06 実行分

### 1. 列名が既知仕様と一致しない（実害なし・情報として記録）
実際のCSV（`日次勤怠一覧_月末締め（翌月末日払い）_2026_07.csv` / `_2026_08.csv`）には、
Routineプロンプトに書かれた「既知仕様」の20列に加えて、以下の列が存在した。
既知仕様の列はすべて部分集合として存在していたため、分析処理自体は問題なく実行できた。

追加で存在した列：`申請アラート`, `勤務・賃金`, `勤務予定開始時刻`, `勤務予定退勤時刻`,
`遅刻早退時間`, `控除対象の遅刻早退`, `有休時間 - 半休`, `有休時間 - 時間休`,
`有給特休時間 - 合計/半休/時間休`, `無給特休時間 - 合計/半休/時間休`,
`始業時刻1〜3`, `終業時刻1〜3`, `休憩1〜3入り/戻り時刻`, `出勤登録種別`, `退勤登録種別`,
`出勤打刻変更前時刻`, `退勤打刻変更前時刻`, `出勤打刻変更後時刻`, `退勤打刻変更後時刻`。

**特に重要**：`勤務・賃金`列に「フレックスタイム」「シフト制」の値が入っており、
Routineプロンプトが言及する「シフト制の従業員」の判定に使える列であることを確認した
（値が`シフト制`の行 = シフト制従業員）。今回時点でシフト制は「大橋 海」1名のみ。

### 2. 打刻漏れ（項目1）の判定ロジック修正が必要
既知仕様書の「出勤時刻・退勤時刻が空欄かどうかで打刻漏れを判定する」を素直に実装すると、
`勤務日種別=所定労働日`の日に有休（`有休`）・半休・趣味休暇・欠勤を取得した日も
「出勤時刻・退勤時刻が空欄」になるため、大量の誤検知（該当者が55名に膨れ上がった）が
発生した。

**対策**：`勤怠種別`列が`未入力`の行のみを「真の打刻漏れ」とすること。
`勤怠種別`が`有休`/`有休（半休）`/`趣味休暇`/`趣味休暇（半休）（半休）`/`欠勤`等の行は、
出勤・退勤時刻が空欄でも打刻漏れではない（正しく休暇として処理済みのため）。
この修正後、該当者は22名になった。今後もこの条件（`勤務日種別=所定労働日` かつ
`勤怠種別=未入力`）を使うこと。

### 3. `data/monthly_history.json`の従業員キーが2種類混在していた（重要・修正済み）
既存の`monthly_history.json`は、CSV方式に切り替える前（freee API方式時代）に作成された
ものと思われ、従業員キーがfreee内部の`employee_id`（例：`1943218`）になっていた。
一方、CSVの`従業員番号`列は会社の社員番号（例：`00002`）であり、**両者は全く別の
ID体系**である。この不一致に気づかずCSVの`従業員番号`をキーとして追記すると、
同一人物のレコードが2つのキーに分裂して蓄積されてしまうバグがあった
（実際に一度発生させかけたため、`git checkout`で該当ファイルを一度リバートして
やり直した）。

**対策**：氏名（空白除去して正規化）を突き合わせキーとして、旧`employee_id`キーの
レコードをCSVの`従業員番号`キーに移行してから、以降は`従業員番号`のみをキーとして
使うように統一した。今後、新規入社者はCSVの`従業員番号`でそのまま追加すればよい。

### 4. `total_excess_statutory_work_mins`の算出元列が間違っていた（重要・修正済み）
既存の`monthly_history.json`の`total_excess_statutory_work_mins`は、実際には
`時間外労働時間`列ではなく`法定内残業時間`列の合計で計算されていたことが判明した
（例：小川陽暉さんの2026年6月は`法定内残業時間`合計316分だが`時間外労働時間`合計は0分。
両者は別概念で、36協定・過労死ライン（項目8・9）の判定に使うべきは`時間外労働時間`
（＋`法定休日労働時間`）であり、`法定内残業時間`（所定労働時間超だが法定8時間以内）
ではない）。

**対策**：`total_excess_statutory_work_mins`は`時間外労働時間 + 法定休日労働時間`の
月間合計（分）として算出し直すよう修正した。既存5名分の2026-06データも含めて再計算し、
過去の誤った値を上書きした。項目9（複数月平均80時間リスク）は今後このフィールドを
根拠に判定すること。

### 5. ステップ0(a)鮮度チェックと月またぎ期間の関係（判明・解決済み）
対象期間が月をまたぐ場合（例：6/29〜7/5）、支給月ベースで2つのCSVが必要になるが、
このうち「実労働月が既に終了・確定している側」のファイル（今回で言う`_2026_07.csv`
＝6月分）の`git log`上の最終コミット日時が前日（7/5）のままで、本日（7/6）の
コミットが存在しなかった。実行時点ではこれを「6月は締まった月なので日々の
再コミット自体が発生しない」と解釈したが、運用担当（國方さん）に確認したところ、
より正確な原因が判明した。

`data/manual_csv/move_log.txt`（Windowsタスクスケジューラの`move_csv.bat`が出力する
ログ。文字コードはUTF-8ではなくShift_JIS/CP932。詳細は本ファイル末尾の項目参照）を
`encoding='cp932'`で正しくデコードすると、本日20:17時点で**`_2026_07.csv`と
`_2026_08.csv`の両方**がDownloadsフォルダから正常に検出・移動されていたことが
分かった：

```
2026/07/06 20:17:06.98
C:\Users\Kunikata\Downloads\日次勤怠一覧_月末締め（翌月末日払い）_2026_07.csv
C:\Users\Kunikata\Downloads\日次勤怠一覧_月末締め（翌月末日払い）_2026_08.csv
        2 個のファイルを移動しました。
```

つまりCowork（freee側のエクスポート）・タスクスケジューラ（ファイル移動）の両工程は
2ファイルとも正常に実行していた。しかしその後の`git commit`では、実際にコミットに
含まれたのは`_2026_08.csv`と`move_log.txt`の2ファイルのみで（`git show --stat`で確認
済み）、`_2026_07.csv`は含まれていなかった。これは**6月分の再エクスポート内容が
既存コミット済み内容とバイト単位で完全に同一だったため、gitが差分なしと判断し
コミット対象にしなかった**ことによるものと考えられる（6月は締まった月のため、
何度エクスポートしても中身が変わらない）。

**結論**：ステップ0(a)の「`git log`上の最終コミット日時が本日か」という判定方法は、
「エクスポート・移動処理が今日実行されたか」の代理指標としては**不完全**である。
内容が変化しない（＝コミットが発生しない）ケースを「未実行」と誤判定しうる。
今回の6月分ファイルについては、実際にはパイプラインは正常に動作しており、
単に内容差分がなかっただけと判明したため、「⚠️ データ取得異常」として扱わなかった
今回の判断は結果的に正しかった。

**今後の運用**：この事象そのものは異常ではないため、Routine側のロジック変更は
必須ではない。ただし、より正確な判定をしたい場合は、`git log`の代わりに
`move_log.txt`（cp932でデコード）を読み、当日分のログに対象ファイル名が
「移動しました」の対象として記載されているかを見る方法に切り替えることを検討する
（ただし追加のパース処理が必要になりクレジット消費が増えるため、費用対効果を
踏まえて判断すること。現状の「現在進行中の暦月のファイルのみ本日コミットを要求する」
という簡易ロジックのままでも実害は出ていない）。

### 6. シフト制従業員（大橋 海）は打刻漏れ判定も不正確な可能性がある
既知の注意点として、シフト制従業員は休日判定（項目4・5・7）が不正確になりうる旨が
仕様書に明記されているが、今回の実行で、シフト制の唯一の対象者「大橋 海」さんの
打刻漏れ（項目1）該当日数が19日と、他の従業員と比べて突出して多いことが判明した。
該当日のデータを見ると、`出勤時刻`が33:30（翌9:30）など日付を跨ぐ勤務が記録されて
いる一方、他の多くの日は完全に空欄（`勤怠種別`も`未入力`）になっており、シフト制の
勤務形態では固定勤務者と異なる打刻方法（実際に勤務がない日でも`勤務日種別=所定労働日`
のテンプレート行が残る等）が使われている可能性がある。

今回は仕様書の指示通り項目1は全員分をそのまま列挙したが、シフト制従業員について
項目1も「要確認」の注記が必要かどうかは、今後の実行状況を見て運用担当と相談し、
必要であれば仕様に追記すること。

### 7. `move_log.txt`の文字コード（参考情報）
`data/manual_csv/move_log.txt`（Windowsタスクスケジューラのログ）は、UTF-8として
読むと文字化けする。おそらくShift_JIS（CP932）で書き出されている。今後この
