# freee CSV 勤怠チェック 学習ログ

このファイルは実行のたびに追記していく。次回実行時は必ず先に読むこと。

## 2026-07-05 初回実行

### 1. シフト制従業員は「打刻漏れ」判定（項目1）にも影響する

指示書では「シフト制のため自動判定が不正確な可能性あり」の注記は項目4・5・7のみに
限定されていたが、実際にシフト制従業員（大橋海）のデータを確認したところ、
シフト制の場合は「所定休日」であっても`勤務日種別`が`所定労働日`のまま、
`勤怠種別`が`未入力`（出勤・退勤時刻は空欄）として記録されることが判明した。
これは固定勤務者が休日に`勤務日種別`＝`所定休日`／`法定休日`となるのと異なり、
CSV上で「本当の打刻漏れ」と「シフト上の休日」を区別する手段がない。

→ **対策**：項目1（打刻漏れ）でも、対象者が`勤務・賃金`＝シフト制の場合は
「シフト制のため所定休日が『未入力』と記録されている可能性があり、真の打刻漏れと
区別できません。要確認」の注記を付けて報告すること。省略・除外はしない
（項目1は全員列挙が必須のため）。

現時点でシフト制の従業員は「大橋 海」（従業員番号00088）の1名のみ（2026年6-7月時点）。
今後シフト制従業員が増えた場合も同様に扱うこと。

### 2. `勤怠種別`に想定外の値（二重サフィックス）を確認

2026年7月分CSVで `福島 千晶`（従業員番号00072）の2026/07/30の行に
`勤怠種別`＝`趣味休暇（半休）（半休）`という、`（半休）`が二重に付いた値を検出した
（本来は`趣味休暇（半休）`のはず）。原因はfreee側の入力・エクスポート時の表記ゆれと
推測されるが未確定。出勤・退勤時刻は空欄、総勤務時間は`04:00`。

→ **対策**：`勤怠種別`の既知値リストに完全一致しない値が出た場合は、データ品質メモとして
報告するだけにとどめ（個別に修正・推測しない）、実害（打刻漏れ誤検知等）がないか
値ごとに確認すること。今回はこの1件のみで実害なし。

### 3. `有休`／`趣味休暇`／`慶弔休暇`等は出勤・退勤時刻が空欄でも総勤務時間が0にならない

これは正常な仕様（休暇取得日は所定時間分を勤務したものとしてみなし残業計算等に使う）。
「出勤・退勤時刻が空欄なのに総勤務時間が0でない」という条件だけでデータ品質異常と
判定すると、有休取得日すべてを誤検知してしまう（初回実行時に大量誤検知が発生し、
`勤怠種別`が休暇系（有休/有休（半休）/趣味休暇/趣味休暇（半休）/慶弔休暇）の場合を
除外するよう修正済み）。次回以降のスクリプトでもこの除外条件を維持すること。

### 4. 「当月」の解釈（項目7・8・9）と「過去1か月」の解釈（項目1）

実行日が週の最終日（日曜）と重なる等、対象週の月と実行日の暦月が異なるケースがある。
このとき指示書の「当月」が実行日ベースの暦月（進行中で日数按分が必要）を指すのか、
対象週が属する月（既に締まった月）を指すのか、指示書だけでは一意に決まらない。

→ **今回の判断**：項目7・8・9の「当月」は「対象週が属する月」（＝対象週チェックと
同じ、既に締まった月）として扱う。理由：項目8の「経過日数から按分した着地見込み」は
締まった月では按分係数が1になるだけで害がなく、逆に実行日ベースの進行中月（日数が
浅い）で判定すると、月初はほぼ全員が「休日不足」「残業未達」判定になり実務上ノイズに
なるため。一方、項目1（打刻漏れ）は指示書に独自の窓（「過去1か月・前日以前」）が
明記されているため、これは実行日基準のローリングウィンドウとして扱う（対象週の月に
限定しない）。次回実行時も同じ解釈を踏襲すること。

### 5. `data/monthly_history.json` の初期状態について

初回実行前、このファイルには過去のfreee API検証（`docs/freee_api_notes.md`参照）で
使ったテスト対象5名分の断片データ（2026-06の一部＋不完全な2026-07）が残っていた。
今回のCSVベース運用開始にあたり、全従業員分の2026-06（確定月）データで上書き・拡充し、
不完全だった2026-07のテストデータは削除した。**月次履歴には確定月（締まった月）のみを
保存する方針**とし、進行中の当月データは`monthly_history.json`に永続化しない
（複数月平均が進行中の月によって不当に薄まらないようにするため）。次回実行時も
この方針を維持すること。
